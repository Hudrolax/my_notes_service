# Правила обработки заметок

## 1. Какие файлы сервис видит

Сервис обходит все файлы с расширением `.md` внутри `/data` рекурсивно.

Не обрабатываются особым образом:

- не-Markdown файлы;
- файлы внутри каталогов `.trash`;
- файлы, изменённые менее 5 минут назад.

## 2. Что считается заметкой-предметом

Заметка считается предметом, если во фронтматтере есть:

```yaml
item: true
```

Проверка реализована небуквально:

- значение приводится к строке;
- затем к нижнему регистру;
- затем сравнивается с `"true"`.

Это значит, что будут считаться истинными, например:

- `true`
- `"true"`
- `TRUE`

## 3. Что считается контейнером вещей

Файл считается вложенным в контейнер вещей, если его родительская папка описана одноимённой заметкой с `item: true`.

Пример:

```text
Пригородная 26/
  Комната/
    Шкаф/
      Шкаф.md
      Куртка.md
```

Если в `Шкаф.md` есть `item: true`, то `Куртка.md` считается лежащей в контейнере вещей.

Если одноимённой заметки нет или в ней `item` не равен `true`, дочерние файлы не участвуют в расчёте `path`.

## 4. Как обновляется поле `path`

Поле `path` корректируется только для тех заметок, которые одновременно:

- находятся внутри контейнера вещей;
- сами имеют `item: true`.

Значение строится как цепочка всех папок-контейнеров от корня `/data` до родительской папки файла.

Пример:

```text
/data/Пригородная 26/Сарайка у папы/Коробка КБ2/Деталь.md
```

даёт:

```yaml
path: Сарайка у папы/Коробка КБ2
```

Если текущее значение уже совпадает с вычисленным, файл не переписывается.

## 5. Как обновляются даты

Логика обновления `creation_date` и `modification_date` в коде есть, но в рабочем цикле сейчас отключена.

Если её включить:

- формат даты будет `ДД.ММ.ГГГГ ЧЧ:ММ:СС`;
- `creation_date` берётся из `ctime`, а если он недоступен, из `mtime`;
- `modification_date` берётся из `mtime`;
- после записи сервис восстанавливает исходный `mtime`, чтобы не провоцировать бесконечные обновления.

## 6. Какие файлы удаляются

Сервис удаляет Markdown-файлы, если:

- имя файла начинается с `Без названия`;
- файл не находится в корне `/data`;
- файл не находится внутри `.trash`;
- файл старше 5 минут.

Примеры имён, подпадающих под правило:

- `Без названия.md`
- `Без названия 2.md`

В режиме `FAKE_FILE_WORKING=true` удаление не выполняется физически.

## 7. Требования к фронтматтеру

Безопасная запись работает только для файлов, которые:

- начинаются с первой строки `---`;
- имеют закрывающую строку `---`;
- содержат YAML mapping между разделителями.

Если фронтматтер повреждён, сервис не должен молча переписывать файл. Он завершит обработку этого файла с ошибкой в логах.

## 8. Правила безопасной записи

При изменении фронтматтера сервис обязан:

- сначала полностью прочитать файл;
- полностью провалидировать структуру;
- только потом писать изменения;
- сохранять исходный стиль переводов строк;
- сохранять BOM, если он был;
- при ошибке записи попытаться вернуть исходное содержимое.

Это одно из главных инвариантов проекта. При доработках нельзя упрощать запись до схемы "открыли файл и переписали вслепую".

## 9. Наблюдаемые ограничения

- Путь рассчитывается только для тех веток, где папки соответствуют файловой конвенции контейнера вещей.
- Фронтматтер читается только в первой YAML-секции файла. Всё остальное содержимое не анализируется.
- Обход использует `os.walk`, поэтому большие каталоги будут сканироваться полностью на каждом цикле.

## 10. Что важно не сломать при развитии проекта

- Режим `FAKE_FILE_WORKING` должен покрывать все операции записи и удаления.
- Исключение `.trash` должно сохраняться, если бизнес-правило не меняется явно.
- Cooldown в 5 минут защищает от гонок с ручным редактированием и должен пересматриваться осознанно.
- Документация должна обновляться вместе с изменением файловых конвенций.

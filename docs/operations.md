# Эксплуатация и запуск

## Предпочтительный способ запуска

Проект рассчитан на запуск через Docker Compose:

```bash
docker compose up --build
```

Контейнер:

- собирается из `Dockerfile`;
- использует Python `3.13.4-alpine3.22`;
- запускает `scripts/run.sh`;
- работает от пользователя `www`;
- ожидает код в `/app` и данные в `/data`.

## Текущая схема volume mount

В `docker-compose.yml` используются два монтирования:

- `${APP_PATH}:/app`
- `${DATA_PATH}:/data`

В текущем `.env` это означает:

- код сервиса берётся из `./app`;
- тестовые данные берутся из `./test_data_folder`.

Такой режим удобен для безопасной локальной отладки.

## Переменные окружения

### Для приложения

- `LOG_LEVEL` — уровень логирования приложения.
- `TZ` — временная зона контейнера.
- `FAKE_FILE_WORKING` — если `true`, сервис не записывает файлы и не удаляет их.
- `DEV` — влияет только на сборку dev-зависимостей в образе.

### Для Docker Compose

- `APP_PATH` — путь к каталогу с кодом, который монтируется в `/app`.
- `DATA_PATH` — путь к каталогу с заметками, который монтируется в `/data`.

## Рекомендуемые режимы работы

### Безопасная локальная проверка

Использовать:

- `FAKE_FILE_WORKING=true`
- `DATA_PATH=./test_data_folder`

Так сервис пройдёт весь цикл обработки, но не будет менять реальные файлы.

### Боевая работа

Для реальной записи нужно как минимум:

- выставить `FAKE_FILE_WORKING=false`;
- смонтировать в `DATA_PATH` реальное хранилище заметок;
- убедиться, что структура каталогов соответствует правилам из `docs/rules.md`.

## Логи

Логи пишутся в stdout контейнера. Формат:

```text
<timestamp> - <logger> - <level> - <message>
```

Полезные команды:

```bash
docker compose logs -f app
docker compose up --build app
```

## Что проверять после изменений

1. Контейнер стартует без ошибок импорта.
2. В логах видно сообщение `my_notes_service запущен`.
3. В логах видно сообщение о запуске задачи корректировки параметров.
4. При `FAKE_FILE_WORKING=true` записи и удаления заменяются сообщениями в логах.
5. При `FAKE_FILE_WORKING=false` сервис меняет только ожидаемые файлы.

## Известные операционные нюансы

- `.env` из корня репозитория не монтируется внутрь `/app`; приложение получает настройки через переменные окружения, проброшенные в `docker-compose.yml`.
- В проекте нет healthcheck и graceful shutdown-логики сверх стандартного завершения процесса Python.

# Архитектура `my_notes_service`

## Назначение

Сервис непрерывно обслуживает дерево Markdown-заметок, смонтированное в контейнер по пути `/data`. Текущая версия проекта не предоставляет HTTP API и не хранит состояние в базе данных. Это фоновый воркер, который:

- сканирует каталог с заметками;
- вычисляет корректный `path` для заметок-предметов;
- удаляет служебные файлы `Без названия*.md`;
- пишет результаты обратно в YAML-фронтматтер Markdown-файлов.

## Контекст выполнения

Базовый сценарий запуска:

1. `docker compose` собирает контейнер из `Dockerfile`.
2. Код приложения монтируется в контейнер как `/app`.
3. Каталог с заметками монтируется как `/data`.
4. Скрипт `scripts/run.sh` запускает `python main.py`.

Из-за такого сценария модульные импорты написаны относительно каталога `/app`, а не как пакет `app.*`.

## Компоненты

### `app/main.py`

Минимальная точка входа. Логирует запуск сервиса и стартует задачу `make_actual_item_params()`.

### `app/config/config.py`

Описывает настройки через `pydantic-settings`.

Используемые поля:

- `TZ` — временная зона процесса;
- `FAKE_FILE_WORKING` — флаг безопасного режима без записи и удаления;
- `LOG_LEVEL` — уровень логирования;
- `main_warehouse_path` — устаревшее поле настроек; текущая логика `path` на него не опирается.

### `app/config/logger.py`

Создаёт единый `dictConfig` для корневого логгера. Настройка логирования вызывается автоматически при импорте пакета `config`.

### `app/tasks/item_params.py`

Содержит всю прикладную логику текущего сервиса:

- `is_file_too_young()` — cooldown 5 минут для недавно изменённых файлов;
- `ensure_correct_path()` — корректировка поля `path`;
- `ensure_correct_dates()` — корректировка дат создания и изменения, пока не используется;
- `remove_unnamed_files()` — очистка временных файлов;
- `make_actual_item_params()` — бесконечный цикл обработки.

### `app/utils/utils.py`

Инфраструктурные функции:

- обход дерева каталогов и параллельный запуск обработчиков;
- чтение YAML-фронтматтера из Markdown;
- проверка, является ли заметка предметом;
- определение, лежит ли файл внутри контейнера вещей.

### `app/utils/file_writing.py`

Ключевой модуль безопасной записи. Он:

- читает файл целиком в память;
- валидирует наличие корректного фронтматтера;
- обновляет только YAML-блок;
- сохраняет стиль перевода строк;
- сохраняет BOM;
- при ошибке записи пытается восстановить исходный файл.

## Основной цикл

Каждую минуту сервис выполняет:

1. Полный обход `/data` по всем `.md`-файлам.
2. Для каждого файла попытку привести `path` к фактической цепочке папок-хранилищ.
3. Отдельный обход дерева для удаления файлов `Без названия*.md`.
4. Сон на 60 секунд.

Обход реализован асинхронно с ограничением параллелизма `max_concurrency=64`.

## Модель данных

Сервис ориентирован на Markdown-файлы с YAML-фронтматтером в начале файла:

```md
---
item: true
path: Комната/Шкаф
creation_date: 10.10.2025 22:27:02
modification_date: 10.10.2025 22:27:02
---
Тело заметки
```

На уровне логики проекта значимы поля:

- `item`
- `path`
- `creation_date`
- `modification_date`

## Как определяется контейнер вещей

Сервис использует файловую конвенцию, а не отдельный индекс:

- если файл лежит в каталоге `X/`;
- и рядом с ним в этом каталоге существует файл `X.md`;
- и у `X.md` во фронтматтере указано `item: true`,
- тогда каталог `X/` считается контейнером вещей.

Пример:

```text
/data/Пригородная 26/Комната/Шкаф/
  Шкаф.md        <- item: true
  Куртка.md      <- для этого файла будет вычисляться path=Комната/Шкаф
```

## Вычисление `path`

Алгоритм в `ensure_correct_path()`:

1. Взять родительскую папку файла.
2. Подниматься вверх по дереву до `/data`.
3. На каждом уровне проверять, является ли папка контейнером вещей.
4. Собрать имена всех найденных контейнеров в путь от внешнего к внутреннему.

Для файла:

```text
/data/Пригородная 26/Сарайка у папы/Коробка КБ2/Деталь.md
```

если `Сарайка у папы/Сарайка у папы.md` и `Коробка КБ2/Коробка КБ2.md` содержат `item: true`, получится:

```text
Сарайка у папы/Коробка КБ2
```

Для файла:

```text
/data/Техника/Patrol Y62 Беляш/Провода.md
```

если контейнером является только `Patrol Y62 Беляш`, получится:

```text
Patrol Y62 Беляш
```

## Исключения и защитные механизмы

- Каталоги `.trash` полностью игнорируются при корректировке и очистке.
- Файлы младше 5 минут не обрабатываются, чтобы не мешать ручному редактированию и внешним процессам.
- Ошибка в одном файле не останавливает общий обход: она логируется внутри `walk_through_files()`.
- Безопасный режим `FAKE_FILE_WORKING=true` отключает запись и удаление, но сохраняет прохождение логики и логи.

## Текущее состояние проекта

Архитектура простая и намеренно плоская:

- нет очередей;
- нет БД;
- нет веб-слоя;
- нет автоматических тестов;
- нет отдельного слоя доменных моделей.

Это упрощает сопровождение, но делает критичной точность файловых правил. Любое изменение бизнес-логики должно синхронно отражаться в `docs/rules.md`.

# AGENTS.md

## Назначение проекта

`my_notes_service` — фоновый Python-сервис для обслуживания Markdown-заметок в каталоге данных. Сейчас сервис делает две вещи:

- исправляет поле `path` во фронтматтере заметок-предметов;
- удаляет лишние Markdown-файлы с именем, начинающимся на `Без названия`.

Сервис рассчитан на запуск в Docker и работает как бесконечный цикл, который раз в минуту обходит каталог `/data`.

## Как устроен запуск

- Основная точка входа: `app/main.py`.
- Логирование настраивается побочным эффектом импорта `config` через `app/config/__init__.py`.
- Основная задача: `tasks.make_actual_item_params()`.
- Контейнер запускает `scripts/run.sh`, который выполняет `python main.py` внутри смонтированного каталога `/app`.

Код опирается на то, что:

- приложение доступно внутри контейнера по пути `/app`;
- данные смонтированы в `/data`;
- настройки приходят через переменные окружения контейнера.

## Ключевая бизнес-логика

### 1. Обновление `path`

Функция `ensure_correct_path()`:

- обрабатывает только `.md`-файлы;
- игнорирует файлы в `.trash`;
- игнорирует файлы, изменённые менее 5 минут назад;
- проверяет, что файл лежит внутри контейнера вещей;
- обновляет поле `path`, если оно отсутствует или не совпадает с фактическим путём.

Контейнер вещи определяется так:

- для файла `/data/.../Папка/Файл.md` сервис ищет файл `/data/.../Папка/Папка.md`;
- если у этого файла во фронтматтере стоит `item: true`, папка считается контейнером вещей.

Поле `path` вычисляется как цепочка всех папок-контейнеров от корня `/data` до родительской папки файла. Жёсткой привязки к `Пригородная 26` больше нет.

### 2. Удаление временных файлов

Функция `remove_unnamed_files()`:

- удаляет `.md`-файлы, имя которых начинается с `Без названия`;
- не трогает корень `/data`;
- не трогает каталог `.trash`;
- не удаляет файлы моложе 5 минут;
- в режиме `FAKE_FILE_WORKING=true` только логирует удаление.

### 3. Обновление дат

Функция `ensure_correct_dates()` уже реализована, но в основном цикле отключена. Если включать её обратно, нужно учитывать:

- она переписывает фронтматтер;
- после записи она вручную восстанавливает `mtime`, чтобы не зациклить обновления;
- её нужно включать только после проверки на реальных заметках.

## Структура репозитория

- `app/main.py` — запуск сервиса.
- `app/config/` — настройки и логирование.
- `app/tasks/item_params.py` — основной цикл и прикладные задачи.
- `app/utils/utils.py` — обход файлов и чтение фронтматтера.
- `app/utils/file_writing.py` — безопасное обновление YAML-фронтматтера.
- `scripts/run.sh` — entrypoint контейнера.
- `docs/` — документация по архитектуре, правилам и эксплуатации.
- `test_data_folder/` — локальный каталог для безопасной проверки через Docker Compose.

## Настройки и окружение

Сейчас в проекте реально используются:

- `LOG_LEVEL`
- `TZ`
- `FAKE_FILE_WORKING`
- `DEV`

Для Docker Compose дополнительно используются:

- `APP_PATH`
- `DATA_PATH`

## Правила безопасных изменений

- Любая новая задача, которая пишет в файлы или удаляет их, должна уважать `FAKE_FILE_WORKING`.
- Если задача меняет заметки на диске, нужно сохранять текущую стратегию безопасной записи из `app/utils/file_writing.py`: полная валидация перед записью, попытка восстановления при ошибке, без временных файлов.
- Для фоновых сканеров сохраняйте фильтры по `.trash` и cooldown, если нет очень веской причины их убрать.
- Не меняйте схему импорта (`from config import ...`, `from tasks import ...`) без проверки сценария запуска в контейнере.
- Если меняется логика определения контейнера вещей, обновляйте документацию в `docs/rules.md` и `docs/architecture.md`.

## Как проверять изменения

Предпочтительный способ:

1. Запустить контейнер через `docker compose up --build`.
2. Оставить `FAKE_FILE_WORKING=true`, пока не станет ясно, что логика корректна.
3. Проверить логи и поведение на данных из `DATA_PATH`.

В репозитории пока нет автоматических тестов. Если добавляется нетривиальная логика, лучше вместе с кодом добавлять сценарии проверки или тестовые фикстуры.

## Известные ограничения

- Сервис жёстко привязан к каталогу данных `/data`.
- Вычисление `path` зависит от файловой конвенции: у папки-хранилища должен существовать одноимённый `.md` с `item: true`.
- `README.md` сейчас пустой, поэтому `docs/` и этот файл — основной источник контекста по проекту.
